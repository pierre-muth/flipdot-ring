Intro:
There’s definitely a pattern in my projects: take a slightly unusual display technology… and scale it up until it becomes unreasonable.
This time: flipdots again.
I had a pile of them sitting around, and instead of building yet another rectangular display, I started wondering: what would a circular flipdot display look like?
That question quickly turned into a constraint-driven design exercise—geometry, driving electronics, power, noise—and eventually into a fully battery-powered circular clock.

Initial idea:
Can we imagine a non-standard flipdot matrix? What about a ring?
How big should the ring be so that the gaps due to angles remain acceptable?
Around 300 segments gives less than 2 degrees between each one.
Perfect for a big, fancy clock.
<hand drawing concept image>

Initial idea:
A ring instead of a matrix sounds simple… but geometry quickly becomes the limiting factor.
The key question: How many segments are needed so that the angular gaps remain visually acceptable?
With ~300 segments, the angle between them drops below 2°, which is good enough to make the circle feel continuous.
That gives a natural use case: a large circular clock.

Electronic boards design:
Driving flipdots at this scale is mostly about managing current, timing, and noise.

Electronic boards design:
7-dot segments and FP2800a IC drivers.
Let’s settle on 24 segments per board. This doesn’t fully use the FP2800a capabilities, but it keeps the board size and cost reasonable, and improves the refresh rate.
That still means 12 boards, for a 288 × 7 dot matrix.
Matrix design requires a double diode per dot.
<driving single dot image>
The data bus of each FP2800a needs to be driven for every dot flip. This can result in quite a high data rate if using shift registers—especially in a noisy environment, as flipdots behave essentially like small solenoids.
I went for daisy-chained PIC microcontrollers to drive the two FP2800a per board. Selected: PIC16F15356.
<schematics image>
KiCad sessions were a bit challenging, working with angles. 4-layer boards, board-to-board power and signals. Coil-driving tracks avoid signal lines… and then comes the matrix routing.

PIC Software:
This part turned out to be more subtle than expected.
A tricky 24-byte circular buffer had to be implemented in the PIC. Driving only the flipping dots feature. 
The SPI interrupt code generated by MCC was too slow, so I had to optimize it to reach 100 kHz.
The protocol: only 7 bits of each of the 24 bytes are used; the remaining bits are used for configuration and data checking. This allows:
- configuring the flip duration,
- forcing (or not) a refresh,
- putting the PIC to sleep.
The FP2800a consumes a few mA each. With 24 of them, this is not negligible. So the PIC powers them via a GPIO to save energy when idle.
A 5 ms timer is reset on every received SPI byte. If there is a pause of 5 ms or more, the data is latched and the dots are driven, a sort of  soft framing mechanism.
There is also a check to ensure:
- data is a multiple of 24 bytes,
- the presence of the 0xAA control byte.
This compensates for the lack of a chip-select line and the noisy environment.

Powering and consumption:
I wanted to power everything with a battery to avoid external wires.

Powering and consumption:
One of the goals was to keep the clock completely wireless.
Since updates only happen once per minute, the average power consumption stays surprisingly low, making battery operation realistic.
There is enough space for several cells.
Selected:
- 5× 18650 Li-ion cells
- a 5S BMS
- step-down module (5V)
- step-up module (24V)

Functionalities:
At this point, it stopped being just a display and became an actual clock.
Needed a good low-power ESP32 dev board → selected Xiao ESP32C6.
- WiFi hotspot for setup
- Sync time once per day via NTP
- Night mode: dots updated only every 10 minutes for quieter nights
- Main battery monitoring with automatic shutdown
- Button to change mode
- Demo mode (because the display is quite impressive)

- Connectivity:
  - WiFi hotspot for setup
  - Daily NTP sync
- Power management:
  - Deep sleep most of the time
  - Battery monitoring with safe shutdown
- User interaction:
  - Mode selection button
  - Demo mode (because the display is quite impressive)

The ESP32 stays in deep sleep most of the time to save power.
Battery life is around 40 days.
